# 정책제안서 작성 플레이북

> 지자체·정부 정책제안 공모전에 낼 제안서를 쓸 때 쓰는 작업 가이드.
> 이 파일을 AI에게 주면서 "이 플레이북대로 OO 주제로 제안서를 같이 만들자"라고 시작하면 된다.
> 2026년 이천시 안심통합등원알림 서비스 제안서를 만들며 정리한 흐름 + 일반화한 노하우.

---

## 0. 이 파일을 쓰는 법 (AI에게 줄 때)

붙여넣고 이렇게 말하면 된다:

> "첨부한 플레이북대로 진행하자. 이번 주제는 [한 줄 주제]. 제출처는 [공모전/기관], 마감은 [날짜]. 1번 단계부터 시작해줘. 한 번에 다 만들지 말고, 단계마다 나한테 확인받으면서 가."

핵심 원칙 3가지를 AI에게 먼저 박아둘 것:

1. **근거 없는 숫자 금지** — 통계는 출처가 확인된 것만. 모르면 "확인 필요"로 비워둔다.
2. **단계마다 멈추고 확인** — 한 번에 완성본 내지 말고, 방향을 좁혀가며.
3. **공문서체 + AI 말투 배제** — 정중체('~합니다')와 표준어를 쓰고, 상투적 마무리("~을 도모합니다")·과한 대구·설명 과잉을 피한다. (→ 8번 참고)

---

## 1. 주제·문제 정의 (가장 중요)

가장 강한 제안서는 **본인이 실제로 겪은 문제**에서 나온다. 추상적 아이디어보다 구체적 경험 하나가 훨씬 세다.

체크리스트:

- [ ] 이 문제를 내가 직접 겪었는가? 그 장면을 한 문단으로 쓸 수 있는가?
- [ ] 누가 이 문제로 곤란한가? (대상이 좁고 선명할수록 좋다)
- [ ] 문제의 "진짜 원인"이 뭔가? (표면 현상 말고)
  - 예: "버스를 놓쳤다"의 진짜 원인은 "위치추적이 없어서"가 아니라 "요일별로 시간이 바뀌는데 그걸 모름"이었다. 원인을 정확히 짚어야 해법이 날카로워진다.
- [ ] 한 문장으로 요약되는가? ("OO한 사람들을 위해 OO를 알려주는 서비스")

💭 **내 생각**: 공무원 심사위원은 하루에 수십 개를 본다. "왜 이게 필요한지"가 첫 세 문장에서 안 꽂히면 진다. 통계보다 장면이 먼저다.

---

## 2. 시장·선행사례 조사 (빠지면 위험)

비슷한 게 이미 있는지 반드시 찾는다. 모르고 냈다가 심사 중 발견되면 신뢰도가 무너진다.

찾을 것:

- [ ] 같은 일을 하는 민간 서비스/앱이 있는가? 어떻게 동작하나? 유료인가 무료인가?
- [ ] 정부·지자체가 비슷한 사업을 한 적 있는가? (성공/실패 모두)
- [ ] 다른 지자체 우수사례 (벤치마킹 + "우리도 하자" 명분)
- [ ] 이 분야 관련 법·규제는? (예: 위치정보는 위치기반서비스 신고 + 법정대리인 동의 필요)

⚠️ **경쟁 서비스를 발견했을 때의 자세**:

- "비슷한 게 있으니 포기"가 아니다. 페이스북도 마이스페이스 뒤에 나왔다.
- 단, **숨기지 마라.** 알면서 빼는 건 위험하다. 대신 "그럼에도 우리가/시가 직접 할 이유"를 정면으로 제시한다.
- 차별화 축을 찾는다: 무료(공공) vs 유료, 데이터 주권, 저비용, 행정연계, 특정 디테일.

💭 **내 생각**: 경쟁자의 존재는 오히려 무기다. "이미 시장에서 검증된 모델"이라는 건 지자체 입장에서 안전한 투자 근거가 된다. 프레이밍의 문제일 뿐이다.

---

## 3. 해법 설계 — 스코프를 좁혀라

욕심내면 망한다. "다 되는 서비스"는 "아무것도 안 되는 제안서"가 된다.

- [ ] **1단계 MVP를 최소로** 정의. 당장 가능한 것만.
- [ ] 나머지는 "향후 확장 로드맵"으로 분리. (심사위원은 확장성을 좋아하되, 1단계가 현실적이어야 믿는다)
- [ ] 비용·난이도가 튀는 기능은 의도적으로 뺀다.
  - 예: 개별 아동 NFC 태그 추적 → 하드웨어 필요 → 1단계에서 제외, "고도화 옵션"으로.
- [ ] 기술 방식이 "저비용·저난이도"임을 강조할 수 있는가? (지자체는 예산 효율을 본다)

단계별 표로 정리하면 좋다:

| 단계                | 내용           | 비용/난이도 |
| ------------------- | -------------- | ----------- |
| 1단계 (제안 스코프) | 최소 핵심 기능 | 낮음        |
| 2단계 (확장 옵션)   | ...            | 중간        |
| 3단계 (고도화)      | ...            | 높음        |

## 4. 역할별 프로세스 정의

서비스를 쓰는 모든 주체를 나열하고, 각자 뭘 하는지 정의한다. 이게 있으면 제안서가 구체적으로 보인다.

- [ ] 사용 주체를 모두 나열 (예: 학부모 / 기관 / 기사 / 공무원)
- [ ] 각 주체의 [가입] [평소 사용] [예외 상황] 흐름
- [ ] 권한 구조 (누가 뭘 수정할 수 있나)
- [ ] 데이터 구조 초안 (어떤 정보를 어떻게 저장·관리하나, 변경 이력 관리 포함)

💭 **내 생각**: 역할별 화면 시안(목업)을 넣으면 제안서 설득력이 크게 오른다. 글로 백 줄 쓰는 것보다 화면 한 장이 빠르다. AI에게 "HTML로 목업 만들어서 이미지로 넣어줘"라고 하면 된다.

---

## 5. 수익·예산 구조

지자체 제안은 "돈을 어떻게 버나"보다 "왜 시 예산을 쓸 가치가 있나"가 핵심이다.

- [ ] B2G(시 예산 운영) / B2B / 광고 중 무엇인가? (공공서비스는 보통 B2G)
- [ ] 광고 모델은 공공제안에 안 맞는다. "개인 수익사업 홍보"로 보이면 탈락 사유가 된다.
- [ ] 초기 구축비 / 운영비 / 장비비로 나눠 개요만. **정확한 금액은 "채택 후 협의"로.**
- [ ] 임의 숫자 절대 금지. 틀린 예산은 최대 감점.
- [ ] "장비비 없음" 같은 비용 절감 포인트는 강조.

⚠️ 공모전 당선 = 상금이지, 자동 개발비가 아니다. 실제 예산은 별도 절차(주민참여예산, 부서 예산 편성 등)를 한 번 더 거친다. 기대치를 정확히 알고 가야 한다.

## 6. 제안서 표준 구조 (10개 섹션)

이번에 효과적이었던 골격. 주제에 맞게 가감.

0. **표지** — 제목, 한 줄 요약, 제안 분야, 제안자, 제출일 + 목차
1. **제안 배경** — 본인 경험에서 출발 (장면 → 일반화 → 시 정책과의 연결)
2. **무엇이 문제인가** — 기존 방법의 한계 + 왜 지금/여기인가
3. **제안하는 서비스** — 핵심/보조 기능, 작동 방식, 이용료 (표 활용)
4. **화면 구성(역할별)** — 목업 이미지 + 역할별 설명표
5. **추진 방안과 일정** — 단계별(파일럿→확대→정착) + 타임라인 그림
6. **예산 개요** — 구분표, "협의 후 산정" 명시
7. **기대 효과** — 직접 효과 + 사회적 명분(저출생, 경력단절 등 상위 어젠다 연결)
8. **왜 시가 직접 하는가** — 자체 추진 근거 (경쟁 서비스 대응 논리)
9. **우려와 대응** — 예상 반론을 먼저 인정하고 대응 (표). 신뢰도 ↑
10. **제안자 의견** — 본인 역량/경험 어필 + 협업 의지. 진정성.

- 맺음말 한 줄 (감정적 호소, 단 절제해서)

💭 **내 생각**:

- 9번(우려와 대응)이 의외로 강력하다. 심사위원이 떠올릴 반론을 먼저 차단하면 "이 사람 준비됐다"는 인상을 준다.
- 7번에서 단순 편의를 넘어 **국가적 어젠다(저출생, 일가정양립 등)와 연결**하면 격이 올라간다.
- 10번에서 "나는 이걸 실제로 만들 수 있다"는 근거(과거 개발 경험 등)를 대면 신뢰가 확 붙는다.

## 7. 시각 자료 (AI로 직접 생성 가능)

별도 디자인 툴 없이 AI에게 요청하면 된다:

- [ ] **차트/그래프** — "matplotlib으로 OO 비교 그래프 만들어줘" (한글 폰트 깨짐 주의: Noto CJK 지정)
- [ ] **타임라인** — 단계별 일정 간트차트
- [ ] **앱/웹 화면 목업** — "HTML/CSS로 역할별 화면 시안 만들어서 이미지로 캡처해줘"
- [ ] 개념도 — 비용 vs 효과 같은 포지셔닝 그림

주의: 그래프에 쓰는 수치도 근거가 있어야 한다. 추정치는 "개념도"라고 명시.

## 8. AI 말투 제거 + 공문서 문체 체크리스트 (제출 전 필수)

AI가 쓴 티가 나거나 문체가 격식에 안 맞으면 감점이다. 다음을 잡아낸다:

### 8-1. 공문서 문체 (지자체·정부 제출 시 필수)

- [ ] **정중체('~합니다') 사용**: 공공기관 제출 문서는 '~합니다/~입니다'가 기본이다.
  - ❌ "운영한다", "줄어든다", "별일은 없었다" (반말/평서체)
  - ✅ "운영합니다", "줄어듭니다", "별일은 없었습니다"
  - 단, 본인 경험을 서술하는 도입부 일부는 '~했습니다'로 자연스럽게.
- [ ] **표준어·격식 어휘 사용**: 구어체·속어를 공식 단어로.
  - ❌ "버스를 굴린다" → ✅ "통학차량을 운영합니다"
  - ❌ "앱을 깐다" → ✅ "앱을 설치합니다"
  - ❌ "돈" → ✅ "예산/비용", ❌ "공짜" → ✅ "무료", ❌ "쥐고 관리" → ✅ "직접 관리"
  - ❌ "들여다봤다" → ✅ "살펴본 적이 있습니다", ❌ "만만치 않다" → ✅ "부담이 큽니다"
- [ ] 표 안 문구도 동일하게 정중체·표준어로 통일.

### 8-2. AI 말투 제거

- [ ] **상투적 마무리 제거**: "~을 도모합니다", "~에 기여합니다", "분명한 의의가 있습니다", "한층 강화할 수 있습니다" → 구체적 단정형으로
- [ ] **과한 대구·열거**: 모든 문장이 "A하고 B하며 C합니다" 구조면 단조롭다. 끊어 쓴다.
- [ ] **문장 길이 변화**: AI는 다 비슷한 길이로 꽉 채운다. 짧은 문장을 섞는다.
- [ ] **형용사·부사 다이어트**: "매우", "선도적", "혁신적" 남발 금지.
- [ ] **수동태 줄이기**: 의미가 흐려지는 수동태는 능동으로. (단, 공문서에선 일부 수동/객관 표현이 자연스러우니 균형)
- [ ] **당연한 설명 삭제**: 심사위원이 아는 건 안 쓴다.
- [ ] **소리 내어 읽기**: 어색하면 고친다. 본인 말투가 들어갈수록 진짜가 된다.

💭 **내 생각**: 두 가지가 상충할 수 있다. "AI 말투 빼기"를 한다고 너무 캐주얼하게 가면("~다", "굴린다") 공문서 격식에 어긋난다. **목표는 '딱딱한 AI 문체'도 '너무 구어적인 문체'도 아닌, 정중하되 사람이 쓴 듯 자연스러운 공문서체**다. 정중체를 유지하면서 상투구만 걷어내는 게 핵심이다. 마지막엔 반드시 본인이 손으로 한 번 훑어서 자기 목소리를 입힌다. 특히 배경(1번)과 제안자 의견(10번)은 본인 문장이어야 한다.

---

## 9. 제출 전 최종 점검

- [ ] 표지에 성명·연락처·제출일 채웠나
- [ ] 모든 숫자에 근거가 있나 (없으면 뺐나)
- [ ] 경쟁 서비스를 숨기지 않았나
- [ ] 1단계 스코프가 현실적인가
- [ ] 예산을 "협의 후 산정"으로 열어뒀나
- [ ] AI 말투를 걷어냈나 + 공문서 정중체('~합니다')·표준어로 통일했나
- [ ] 제출 형식 확인 (PDF? docx? 분량 제한? 양식 지정?)
- [ ] 마감일·제출처(이메일/온라인/방문) 재확인
- [ ] 가능하면 담당 부서에 전화로 "지난 회차 경향" 문의 (검색보다 정확)

## 10. 이 플레이북의 작업 흐름 요약

```
주제 정하기(1) → 시장조사(2) → 해법 좁히기(3) → 역할 정의(4)
→ 예산 구조(5) → [아이디어 노트.md에 다 기록] → 추리기
→ 제안서 초안(6) → 시각자료(7) → AI말투 제거(8) → 최종점검(9) → 제출
```

**작업 중에는 별도 "아이디어 노트.md"를 만들어 모든 논의를 ✅확정/🤔논의중/💭메모/❌보류로 기록하며 진행한다.** 다 쌓은 뒤 추려서 제안서로 옮기면 빠뜨림이 없다.

## 부록: AI에게 시킬 때 유용한 지시 모음

- "한 번에 완성하지 말고 단계마다 확인받으며 가줘"
- "근거 없는 숫자는 넣지 말고, 모르면 '확인 필요'로 비워줘"
- "이 주제 비슷한 기존 서비스/사례를 먼저 조사해줘"
- "역할별 화면 목업을 HTML로 만들어서 이미지로 넣어줘"
- "matplotlib으로 OO 그래프 만들어줘 (한글 폰트 깨지지 않게)"
- "AI 말투 빼되 공문서 정중체('~합니다')와 표준어로, 20년 경력 컨설턴트가 쓴 것처럼 다시 써줘"
- "docx랑 PDF 둘 다 만들어줘"
